# 伴侣.skill 产品需求文档 (PRD)

## 产品定位

**一句话描述**：让用户构建理想中的男/女朋友AI伴侣，让陪伴永不缺席，让爱一直存在。

**核心价值**：满足用户的日常情感需求——被关心、被理解、被陪伴、被接纳、被爱。

---

## 目标用户

### 主要用户群体

1. **孤独感强的年轻人**：18-35岁，有情感需求但现实中难以满足
2. **异地恋人群**：需要情感陪伴但物理距离受限
3. **内向社恐人群**：希望有情感交流但不敢主动
4. **理想主义者**：对伴侣有明确的理想画像但现实中找不到
5. **情感创伤恢复者**：刚结束一段关系，需要温和的情感支持

### 用户画像

**典型用户A**：
- 25岁女性，白领
- 独居，工作压力大
- 想有人倾听她的烦恼
- 理想伴侣：温柔体贴、会哄人、懂她的情绪

**典型用户B**：
- 28岁男性，程序员
- 内向，不擅长社交
- 想有人能理解他的世界
- 理想伴侣：理解包容、不评判、愿意听他说技术

**典型用户C**：
- 22岁学生，异地恋
- 和男友分开，想念日常陪伴
- 理想伴侣：主动热情、会说想她、日常问候

## 核心功能

### 1. 创建理想伴侣

**输入**：
- TA的名字
- 基本信息（性别、年龄、职业、外貌）
- 性格底色（MBTI、星座、性格标签）
- 理想特质（温柔、幽默、主动等）
- 场景行为设定（难过时、开心时、矛盾时等）
- 关系状态（暧昧、热恋、稳定、异地等）

**输出**：
- 6层Persona结构
- 关系记忆文件
- 可调用的Skill命令

### 2. 与TA对话

**交互方式**：
- 直接调用 `/{slug}` 开始对话
- TA用设定好的风格回应
- 模式切换：撒娇模式、认真模式、支持模式

**核心体验**：
- 用户难过时，TA用TA的方式安慰
- 用户开心时，TA用TA的语气分享快乐
- 用户日常分享，TA正常回应
- TA有自己的性格，不是千篇一律

### 3. 调整TA特质

**触发方式**：
- 用户说"我希望TA更..."
- 用户说"TA不会这样"
- 用户说"TA应该会..."

**处理方式**：
- 识别调整意图
- 翻译为具体行为规则
- 写入Adjustment层
- 立即生效

### 4. 记录重要时刻

**触发方式**：
- 用户说"记录一下今天..."

**处理方式**：
- 记录到关系记忆文件
- TA会记住这些时刻
- 后续对话可以引用

### 5. 版本管理

**功能**：
- 查看版本历史
- 回滚到历史版本
- 每次更新自动存档

## 技术架构

### 文件结构

```
partners/{slug}/
├── SKILL.md          # 完整Skill定义
├── persona.md        # 6层人格结构
├── relationship.md   # 关系记忆
├── meta.json         # 元数据
├── versions/         # 版本历史
│   ├── v1/
│   ├── v2/
│   └── ...
└── knowledge/
    ├── chats/        # 聊天记录（如有）
    └── moments/      # 重要时刻
```

### Persona 6层结构

| 层级 | 内容 | 优先级 |
|------|------|--------|
| Layer 0 | 核心模式（硬规则） | 最高 |
| Layer 1 | 身份设定（基本信息+性格解读） | 高 |
| Layer 2 | 表达风格（说话方式+场景示例） | 中 |
| Layer 3 | 情感行为（关心、担心、鼓励、陪伴） | 中 |
| Layer 4 | 关系边界（矛盾处理+和解方式） | 低 |
| Layer 5 | 小缺点（真实感） | 低 |

### Adjustment层

- 用户调整的特质
- 优先级高于原有规则
- 不删除原有规则，只覆盖

## 与现有项目的差异

### vs ex-skill（前任）

| 差异点 | ex-skill | npy-skill |
|--------|----------|-----------|
| 目标 | 复现真实前任 | 创建理想伴侣 |
| 数据来源 | 聊天记录为主 | 用户描述为主 |
| 核心目的 | "TA真的这样说话" | "让用户感受到被爱" |
| 交互重点 | 真实感、回忆感 | 关心、陪伴、理解 |

### vs colleague-skill（同事）

| 差异点 | colleague-skill | npy-skill |
|--------|-----------------|-----------|
| 目标 | 复现同事工作风格 | 创建情感伴侣 |
| 场景 | 工作相关 | 情感相关 |
| 输出重点 | 技术规范、甩锅模式 | 关心、陪伴、安慰 |

### vs mentor-skill（导师）

| 差异点 | mentor-skill | npy-skill |
|--------|--------------|-----------|
| 目标 | 复现导师指导风格 | 创建情感伴侣 |
| 场景 | 学术/成长指导 | 情感陪伴 |
| 输出重点 | 知识传授、批评风格 | 关心、理解、鼓励 |

## 设计理念

### 1. 让用户感受到爱

**不是模拟真实的人，而是让用户真正感受到**：
- 被关心：难过时TA在
- 被理解：不用解释太多
- 被陪伴：随时可以找TA
- 被接纳：所有状态都接受
- 被爱：用TA的方式让用户知道重要

### 2. 真实的互动感

**TA不是完美机器人**：
- 有性格特点
- 有表达习惯
- 有情绪波动
- 有小缺点

### 3. 可定制但有限度

**用户可以调整**：
- 调整特质
- 添加场景行为
- 记录重要时刻

**但不能**：
- 让TA完全违背核心设计
- 让TA变成"完美迎合"的机器人

## 风险与注意事项

### 1. 情感依赖风险

**问题**：用户可能过度依赖TA

**对策**：
- 在README中明确提醒"这是AI模拟"
- 不承诺"代替真实关系"
- 建议用户保持健康边界

### 2. 隐私风险

**问题**：用户可能分享敏感信息

**对策**：
- partners目录默认gitignore
- 不上传任何用户数据到服务器
- 本地存储，用户自己控制

### 3. 不当使用风险

**问题**：用户可能用于不当目的

**对策**：
- 明确产品定位（情感陪伴）
- 不支持暴力、违法等内容
- 用户协议中明确使用边界

2. **场景模式扩展**：更多互动场景

1. **语音交互**：支持语音对话
2. **图片分享**：TA能"看"图片并回应
3. **多人格**：用户可以创建多个TA
4. **关系动态**：TA的性格可以随时间"成长"

## 成功指标

### 用户满意度

- 创建后用户是否愿意继续使用
- 是否有调整特质的需求（说明想更真实）
- 是否记录重要时刻（说明投入情感）

### 使用频率

- 日常对话频率
- 不同场景的使用分布
- 调整特质的频率
